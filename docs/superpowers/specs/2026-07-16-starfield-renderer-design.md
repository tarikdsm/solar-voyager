# Starfield renderer design

## Goal

Render the complete 9,096-entry Yale Bright Star Catalog as a stable,
photometrically ordered background at every supported camera translation and
zoom. The renderer must consume the already validated `stars.bin` payload,
preserve its ecliptic J2000 directions and B−V-derived colours, and add exactly
one draw call.

## Chosen representation

The starfield is one `THREE.Points` backed by one `BufferGeometry` and one
`ShaderMaterial`. One mesh or DOM node per star would violate the draw-call and
setup-cost budgets. `PointsMaterial` cannot express independent per-star point
size and opacity, so a small dedicated shader is required.

The catalog's existing seven-float payload is attached without copying through
one `InterleavedBuffer`:

- `position`: unit ecliptic direction, components 0–2;
- `aMagnitude`: visual magnitude, component 3;
- `aColor`: baked display RGB, components 4–6.

Two setup-only `Float32Array` attributes hold the derived display size and
opacity. No geometry, material, typed array, or temporary object is created or
mutated in the frame loop. The points object has `matrixAutoUpdate = false` and
`frustumCulled = false`.

## Apparent size and brightness

The visual flux proxy for magnitude `m` is `F = 10^(-0.4m)`. At setup, each
star receives:

```text
sizeCssPx = clamp(1, 4, 1 + 1.5 * F^0.25)
opacity   = clamp(0, 1, 10^(-0.4 * (m - 1)))
```

The vertex shader multiplies CSS size by the startup device pixel ratio. This
keeps the apparent footprint stable across the renderer's supported pixel
ratios while preserving the catalog's brightness ordering. Stars at or below
the unresolved footprint use a square one-fragment point; resolved stars use a
soft circular point-spread profile. Additive alpha blending lets bright stars
read naturally without creating per-star textures.

The RGB attribute is consumed unchanged. It is already the bounded display
mapping of Yale B−V generated by `tools/bake_stars.py`; missing colour indices
are neutral white. T0081 may transform the same colour and direction attributes
for relativistic Doppler and aberration without changing the catalog schema.

## Camera and depth contract

The shader places each unit direction on a `1e9 km` sphere. Because
`CameraRelativeSpaceScene` keeps the Three.js camera permanently at the origin,
the object remains at the render origin and is deliberately not registered as
a physical-body binding. It therefore has no translational parallax and needs
no per-frame update, even across arbitrary heliocentric warp jumps.

After projection, the vertex shader assigns `gl_Position.z = gl_Position.w`,
placing stars on the far plane. The material uses a less-or-equal depth test so
that cleared far-plane pixels pass, but disables depth writes. Opaque planets consequently occlude the background even though
their real camera-relative distance can exceed the nominal star-sphere radius.
The starfield itself never hides later transparent effects by populating depth.

Zoom changes only the projection matrix. Catalog directions remain exact and
the point footprint remains expressed in pixels, so changing vertical field of
view changes angular framing without inflating individual stars.

## Ownership and startup flow

`Starfield` owns the Three.js buffers, geometry, material, and points object. It
offers setup-time creation, a device-pixel-ratio uniform update for renderer
configuration changes, and `dispose()` for deterministic resource teardown.
It exposes no frame update method.

`createEpochWorld()` loads and validates `data/stars.bin` before shader
precompilation, constructs the starfield, and returns it with the world. Tests
may inject a validated catalog so unit tests do not perform I/O. A missing,
truncated, non-finite, out-of-range, or non-unit catalog remains a loud startup
failure through `loadStarCatalog()` / `parseStarCatalog()`; the renderer does
not substitute synthetic stars.

## Verification

- Unit tests lock the one-points/one-material topology, zero-copy interleaved
  catalog attributes, exact magnitude mapping, B−V RGB passthrough, fixed
  radius, far-plane depth, setup-only matrices, pixel-ratio updates, and
  disposal.
- Epoch-world tests prove the real catalog is integrated before the only shader
  compile and remains separate from physical body bindings.
- A browser regression points the camera at seven catalog-indexed Orion stars:
  Rigel, Bellatrix, Mintaka, Alnilam, Alnitak, Saiph, and Betelgeuse. It verifies
  luminous pixels at their CPU-projected locations and the three-star belt
  alignment.
- The browser regression repeats after large synthetic heliocentric camera
  translations and at multiple fields of view. Equal pixels for translation
  and correct projected detections for zoom establish warp and zoom stability.
- Full lint, typecheck, Vitest, production build, task-schema, asset-budget, and
  render benchmark gates run before review.
